# Plan: マルチカメラ extrinsic calibration (issue #9 / Phase E)

## ゴール

複数 Femto Bolt が同じワールド空間に揃うよう、各カメラの `global_tr_colorCamera`（4x4 transform）を求める Editor ツールを実装。
結果を `<root>/calibration/extrinsics.yaml` に書き出し、`PointCloudRecorder.Read` で読み戻して点群が世界座標で重なるようにする。

完了条件は GitHub issue #9 を参照。

## アーキテクチャ概要

```
[ChArUco / AprilTag ボードを物理空間に置く]
   ↓
各 PointCloudRenderer の color フレームから marker を検出
   ↓
各カメラ視点で marker pose (camN_tr_marker) を求める
   ↓
基準カメラ（index 0）を world とする
world_tr_camN = (cam0_tr_marker) * (camN_tr_marker)^-1
   ↓ ※ 表記・導出は「Transform 規約」節参照
extrinsics.yaml の各 device エントリに `global_tr_colorCamera` として書き込み
   ↓
Read 時に読み戻して、live=Manager / playback=Recorder の各 GO transform に反映
```

## Transform 規約（**プロジェクト唯一の正**）

混乱防止のため、ここで一度だけ規約を定義し、コード・YAML・コメント・テストすべてこれに従う。

### 表記とチェイン
- 表記: `A_tr_B` は「**B 座標系の点を A 座標系へ変換する** 4x4」
- 単位: 並進は **メートル**。回転は 3x3 row-major (YAML では rotation: 9 要素配列)
- world 系の取り方: **基準カメラ (index 0) の color camera 系を world** とする → `global_tr_colorCamera[0] = identity`
- world 統合チェイン: marker 経由
  - `marker_tr_camN = (camN_tr_marker)^-1`
  - `world_tr_camN = world_tr_marker * marker_tr_camN = global_tr_colorCamera[0] * (cam0_tr_marker) * (camN_tr_marker)^-1`
  - 基準が identity なので実装上は `global_tr_colorCamera[N] = cam0_tr_marker * (camN_tr_marker)^-1`

### 座標系（**OpenCV camera convention で統一**）

- **calibration 値の格納形式 = Orbbec / OpenCV camera convention**: +x right, +y down, +z forward, 右手系, meters
  - 理由: Orbbec SDK の D2C extrinsic と OpenCV ArUco の `cv::solvePnP` 出力がどちらもこの系。マーカー pose 推定の自然な出力なのでここを正とすることで basis 変換が消える
  - `extrinsics.yaml` のヘッダコメントもこれに合わせて更新する（既存 writer の "+x left, +y up, +z forward" は誤記なので修正対象）
- **Unity scene = +x right, +y up, +z forward, 左手系**
- **現在の実行時 basis 変換**: `PointCloudRenderer.cs:594-597` で `transform.localScale.y = -|y|` のみ。これは「OpenCV camera (+y down) → Unity (+y up)」の Y 反転。X / Z は変えない
- **`ExtrinsicsToUnityMatrix(ObExtrinsic ocv)` の責務**:
  - 入力: OpenCV camera convention の 4x4
  - 出力: Unity 左手系で同等の transform を表す 4x4 = `S * M_ocv * S^-1` (`S = diag(1, -1, 1)`)
  - 物理意味: Y 反転を世界座標基底変換として正しく相似変換で適用する（並進ベクトルの y も反転、回転軸の y 成分も整合）
- **適用ポイント** (重要): `global_tr_colorCamera` を Unity の transform に焼くときは `ExtrinsicsToUnityMatrix` を必ず通す。**既存の `localScale.y = -1` (renderer / playback の点群 GO 内部の Y 反転) はそのまま残す**。これは「点群ローカルメッシュ点 (OpenCV) → Unity 表示」の per-renderer 変換であり、`ExtrinsicsToUnityMatrix` は「world ← color camera」の上位 transform の basis 変換であって、責務が直交している
- **basis 変換テスト**: Phase 3 のテストに「OpenCV 既知 transform → `ExtrinsicsToUnityMatrix` → Unity 上の点位置が想定どおり」を追加（X 鏡像にならないこと）

### 検証
Phase 3 に「合成 (synthetic) 入力で既知の真値と比較する EditMode テスト」を含める（後述）：
- (a) チェイン算術 (逆行列、合成順、row/col-major)
- (b) basis 変換 (`ExtrinsicsToUnityMatrix` で X 鏡像にならないこと)

## YAML スキーマ（**既存 writer に合わせて凍結**）

`Assets/Scripts/PointCloud/PointCloudRecording.cs` L305〜の出力フォーマットを正とする。Plan 側は新規 schema を作らない。**ヘッダコメントの座標系記述だけ "OpenCV camera convention (+x right, +y down, +z forward), meters" に修正する** (上記 Transform 規約と整合)。

```yaml
# Generated by FloatingVectorsICC. Calibration values are expressed in the
# Orbbec/OpenCV camera convention: +x right, +y down, +z forward (right-handed),
# translations in meters. Unity-side basis conversion happens at apply time.
cameras:
  - device_serial: "CL8F253004Z"
    color_intrinsic:
      fx: ...
      fy: ...
      cx: ...
      cy: ...
      width: 1280
      height: 720
    depth_intrinsic: { ... }
    depth_distortion: { ... }
    color_distortion: { ... }
    colorCamera_tr_depthCamera:    # SDK D2C (depth → color), meters
      rotation: [r00..r22]
      translation: [tx, ty, tz]
    global_tr_colorCamera:         # NEW: world ← color camera, meters。基準カメラは identity
      rotation: [r00..r22]
      translation: [tx, ty, tz]
```

トップレベルキーは `cameras:` (NOT `devices:`)。フィールド名は既存 writer と完全一致させる。

## 設計判断

### Marker フォーマット
- **ChArUco** (Charuco = Chessboard + ArUco) を推奨：チェッカーで sub-pixel 精度、ArUco で個別マーカー識別
- A4 〜 A2 印刷で十分（精度は紙のフラットさに依存するので硬い板に貼ると良い）
- 参考: `cv::aruco::CharucoBoard` の dictionary は `DICT_4X4_50` あたりで十分

### Marker 検出ライブラリ
3 つ選択肢：

1. **OpenCV for Unity** (Asset Store 有料): 最も簡単、Unity ネイティブ
2. **OpenCVSharp** (NuGet): Windows ネイティブ、P/Invoke 経由で Unity から使える
3. **自前 ArUco 実装**: 実装コスト大、推奨しない

最初は 2. OpenCVSharp が依存最小・ライセンス自由・Windows x64 で問題なし。

### 撮影タイミング & timestamp スキュー検査
- ユーザーが「Capture」ボタンを押した瞬間に全カメラの最新カラーフレームをスナップ
- `PointCloudCameraManager` 経由で全 `Renderer.OnRawFramesReady` の最新カラーバッファ + **device timestamp (μs)** を保持
- **スキュー gating はキャプチャセット単位 (atomic)**: 参加カメラの color frame timestamp の `(max - min)` が `maxSkewMs` (Inspector 既定 16ms ≒ 1 フレーム @ 30Hz) を超えたら**そのキャプチャセット全体を破棄**。1 台でも閾値超過なら他カメラのフレームも採用しない（per-device reject にすると不揃いペアが calibration を汚染するため）
- HW sync 構成 (issue #6 Phase A) 下では数 ms に収まる想定
- マーカーが全カメラで同時に見える必要があるので、撮影位置を動かしながら複数枚撮ってベストショットを採用する設計

### Live と Playback の transform 反映分離
- **Live** (PointCloudRenderer 駆動): `PointCloudCameraManager.applyExtrinsics` トグルが ON なら、各 spawned renderer GO の `transform` を `global_tr_colorCamera` から構築した `Matrix4x4` で設定
- **Playback** (PointCloudRecorder.Play 駆動): recorder 配下に作る playback 子 GO (`PointCloudRecorder.cs:527` 周辺) の `transform` を、Load 時にパースした per-track `global_tr_colorCamera` で設定
- 両経路は別関数 `ApplyExtrinsicsToLive(Renderer)` / `ApplyExtrinsicsToPlaybackTrack(playbackGO, track)` として実装し、共通の `ExtrinsicsToUnityMatrix(ObExtrinsic)` ヘルパで Unity 座標 (右手→左手 / mm→m など必要な変換) に揃える

## 段階的実装

### Phase 1: 依存追加（半日）
1. **OpenCvSharp**:
   - 取得: `OpenCvSharp4` 4.10.0.20240616 + `OpenCvSharp4.runtime.win` 4.10.0.20240616 を **NuGetForUnity** 経由 (推奨) または手動 DLL 配置
   - 配置先: `Assets/Plugins/OpenCvSharp/` (Windows x64 only にプラットフォーム絞り)
   - **Plugin import 設定**: `OpenCvSharpExtern.dll` は `CPU = x86_64` / `OS = Windows` のみチェック (他プラットフォームの自動 import を切る)
   - DLL 欠損時の fallback: Calibration ウィンドウ起動時に `Type.GetType("OpenCvSharp.Cv2, OpenCvSharp")` で存在確認、無ければ「OpenCvSharp が未配置」とウィンドウに表示して機能を disable
2. **YamlDotNet**:
   - 取得: `YamlDotNet` 16.x を NuGetForUnity 経由
   - パース時は **safe mode**（型自動 deserialization 無し: `DeserializerBuilder` で `WithTypeConverter` を使わず素の `YamlStream` でノード走査）
3. **バージョン pinning と checksum**:
   - `Packages/manifest.json` または NuGetForUnity の `packages.config` に **exact version** を記録
   - native DLL の SHA-256 を `Assets/Plugins/OpenCvSharp/CHECKSUMS.txt` に記載
4. **`.gitignore` 確認**: `Assets/Plugins/OpenCvSharp/` を含める方針なら `.gitignore` から除外、含めないなら手順書化
5. **動作確認**: 起動時に `Cv2.GetVersionString()` を Console に出力する smoke test スクリプト

### Phase 2: 1 カメラの marker pose 推定（1 日）
1. `Assets/Scripts/Calibration/CharucoBoardSpec.cs` に board 仕様（squares X/Y, square length, marker length, dictionary）を ScriptableObject で持つ
2. `Assets/Scripts/Calibration/MarkerPoseEstimator.cs`:
   - 入力: RGB8 byte[], width, height, intrinsic, distortion
   - OpenCvSharp で `Mat` を作って ArUco 検出 → `EstimatePoseCharucoBoard`
   - 出力: rvec, tvec から組み立てた 4x4 transform `cam_tr_marker`（color camera 系で見た marker）
3. テスト: 1 台の Femto Bolt の color フレームを Editor 上で取得して board 検出できることを確認

### Phase 3: 複数カメラの pose を world 統合 + 数学テスト（1 日）
1. `Assets/Scripts/Calibration/MultiCameraCalibrator.cs`:
   - 各 Renderer から最新カラーフレーム + device timestamp を取得
   - timestamp スキュー > `maxSkewMs` のサンプルは reject
   - 全カメラで marker 検出に成功したサンプルだけ採用
   - 基準カメラ (Inspector で指定 / default index 0) の color camera 系を world とする
   - 各 N について `global_tr_colorCamera[N] = cam0_tr_marker * (camN_tr_marker)^-1`
2. **EditMode テスト** `Assets/Tests/Editor/CalibrationMathTests.cs`:
   - 既知の `world_tr_marker`, `world_tr_camN` を合成
   - そこから生成した `camN_tr_marker` を入力としてチェイン計算 → 元の `world_tr_camN` と一致するか assert
   - これで「marker 経由のチェイン」「逆行列の取り方」「row/col-major 取り違え」を catch
3. 複数サンプル取得で平均 / 外れ値除外したい場合は次フェーズで対応、まずは 1 サンプルで動くこと

### Phase 4: extrinsics.yaml の保存と読み込み（1 日）
1. **Writer 改修なし** — `PointCloudRecording.WriteExtrinsicsYaml` (L305〜) は既に `global_tr_colorCamera` を吐く構造。Calibration 結果を `DeviceCalibration.GlobalTrColorCamera` に詰めて呼ぶだけ
2. `PointCloudRecording.ReadExtrinsicsYaml(string rootDir)` を新設:
   - 戻り値: `IReadOnlyList<DeviceCalibration>` (serial / intrinsics / distortion / D2C / global)
   - パーサ: YamlDotNet の `YamlStream` でノード走査 (型自動 deserialization 無し)
   - **必須フィールドが欠けたら例外** (silent fallback 禁止) — 行番号と一緒にメッセージ
   - **path safety**: 入力 path は `<root>/calibration/extrinsics.yaml` 固定。`..` 等を含む device_serial は escape 済みでも reject
3. `PointCloudRecorder.Load`:
   - RCSV 読み込み後に `ReadExtrinsicsYaml` を呼ぶ
   - **per-track 画像寸法 (`DepthWidth/Height`, `ColorWidth/Height`) は RCSV センサーヘッダから読む** — yaml の intrinsic.width/height は calibration 時の値で、録画時にモード変更があると drift する可能性があるため真のソースは RCSV ヘッダ
   - yaml の intrinsic / distortion / D2C / `global_tr_colorCamera` は **calibration 専用** として track の `CameraParam` (intrinsics/distortion/D2C) と `GlobalTrColorCamera` に格納 (offline reconstruction 対応、live 時の renderer に依存しない)
   - yaml が無ければ identity + 警告ログ (旧 dataset 互換)
   - 寸法不整合 (RCSV vs yaml intrinsic.width/height が極端に違う) はログで warning
4. **Round-trip テスト** `Assets/Tests/Editor/ExtrinsicsYamlRoundTripTests.cs`:
   - DeviceCalibration を組んで Write → Read → 値一致 (回転 / 並進の epsilon 比較)

### Phase 5: ランタイム反映（半日）
1. `PointCloudCameraManager` に `bool applyExtrinsics` トグル追加
2. **Live**: `applyExtrinsics` が ON のとき、spawn 直後 / extrinsics 再読込時に各 Renderer GO の transform を `global_tr_colorCamera` から計算して設定
3. **Playback**: `PointCloudRecorder.Play` で playback GO を生成する箇所 (`PointCloudRecorder.cs:527` 周辺) で、Load 時にパースした per-track `global_tr_colorCamera` を transform に設定
4. 共通 `ExtrinsicsToUnityMatrix(ObExtrinsic ocv)` を実装 — **入出力は「Transform 規約 → 座標系」節の定義に厳密に従う**：
   - 入力: OpenCV camera convention の 4x4 (`+x right, +y down, +z forward`, 右手系, meters)
   - 出力: Unity 左手系で同等の transform を表す 4x4 = `S * M_ocv * S^-1` (`S = diag(1, -1, 1)`)
   - 単位は格納時点で meters なので追加スケーリング不要 (mm→m は SDK 値を `DeviceCalibration` に詰める段階で済ませる)
   - per-renderer `localScale.y = -1` は触らない（責務分離）
5. 点群が物理空間で重なるかを目視確認

### Phase 6: Editor UI（半日 〜 1 日）
1. `Assets/Scripts/Calibration/Editor/CalibrationWindow.cs` を新設
2. メニュー: `Window > Calibration > Multi-Camera Extrinsic`
3. UI:
   - Board spec 選択（ScriptableObject）
   - `maxSkewMs` 入力欄
   - 「Capture」ボタン: 全カメラの最新フレーム + timestamp を取って pose 推定 → サンプル蓄積（skew reject は表示）
   - サンプルリスト表示（カメラ N が見えた / 見えなかった、skew ms）

   - **Re-projection error** を per-camera 表示（quality check 用）
   - 「Reset」ボタン: identity に戻す（書き出し済みも上書き）

## 既知の制約 / 注意

- **同時撮影**: 全カメラで同時にマーカーが見える必要がある。HW sync 構成（issue #6 Phase A 完了）の下では `maxSkewMs` 数 ms 内に収まる想定
- **物理セットアップ**: マーカー板はフラット（曲がると歪む）、十分な照明、各カメラから 1〜3 m 程度の距離
- **対称性**: ChArUco は対称デザインだと姿勢推定が反転することがある。非対称 dictionary 推奨
- **CLAUDE.md ルール**:
  - v2 ヘッダ参照必須 — `C:\dev\OrbbecSDK_v2\include\libobsensor\h\` の `ObTypes.h` 等
  - K4A Wrapper 経路は使わない（calibration は color フレームベース、k4abt は無関係）
  - 動作検証は AI 自身が MCP + Editor.log で進める（CLAUDE.md L29 以降）
- **依存配置**: OpenCvSharp / YamlDotNet の DLL 配置場所と pinning 方針を CLAUDE.md にも追記する
